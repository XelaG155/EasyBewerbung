### Rollback-Befehle (vollständig)

Wichtig: der Deploy-Pipeline läuft **kein** `alembic upgrade` — stattdessen legt `init_db()` in `app/database.py` via `Base.metadata.create_all` die Tabellen an. Ein einfaches Git-Rollback reicht deshalb fast, **aber** die neuen Tabellen bleiben auf Disk, bis du sie manuell wegmigrierst.

```bash
# Option A: Code zurück, Tabellen aufräumen (empfohlen wenn schon in Prod)
cd /home/alexgiss/EasyBewerbung
git checkout main
git branch -D refactor/admin-forms-stage1-2
# neue Tabellen explizit wegräumen (nach git revert / bevor redeploy):
docker compose exec easybewerbung-backend alembic downgrade 20260104_02
docker compose up -d --build

# Option B: nur Branch verwerfen (solange noch nicht deployed)
git checkout main
git branch -D refactor/admin-forms-stage1-2

# Option C: auf den Backup-Tag zurück
git reset --hard backup/pre-form-refactor-2026-04-05

# Option D: komplettes Repo aus Tarball wiederherstellen
cd /home/alexgiss
mv EasyBewerbung EasyBewerbung.broken
tar -xzf /mnt/data_disk/backups/easybewerbung/pre-form-refactor-2026-04-05.tar.gz
```

**Warnung zu Option A:** Wenn im Pilot schon `DocumentType`- oder `LlmModel`-Rows vom Admin angelegt wurden, gehen die beim Downgrade verloren. Vorher `pg_dump -t document_types -t llm_models easybewerbung > backup.sql` ausführen.

## Deploy-Anmerkungen

- Dockerfile-CMD ist `uvicorn app.main:app` — kein Alembic-Aufruf.
- `init_db()` im FastAPI `@app.on_event("startup")` nutzt `Base.metadata.create_all` → neue Tabellen entstehen auto bei Redeploy.
- `init_db()` seedet nach diesem Refactor zusätzlich `document_types` und `llm_models` idempotent (nur wenn leer).
- Nach dem Deploy: erster Seitenaufruf an `/admin/documents` triggert `loadAll()`, der Admin sieht sofort die befüllten Tabellen.
