Der Deploy-Pipeline läuft **kein** `alembic upgrade` — stattdessen legt `init_db()` in `app/database.py` via `Base.metadata.create_all` die Tabellen an und seedet sie idempotent.

```bash

cd /home/alexgiss/EasyBewerbung
git checkout main
git branch -D refactor/admin-forms-stage1-2
# neue Tabellen explizit wegräumen:
docker compose exec backend alembic downgrade 20260104_02
docker compose up -d --build

# Option B: nur Branch verwerfen (solange noch nicht deployed)
git checkout main
git branch -D refactor/admin-forms-stage1-2

# Option C: auf den Backup-Tag zurück

# Option D: komplettes Repo aus Tarball wiederherstellen
```

**Warnung zu Option A**: Wenn im Pilot bereits `DocumentType`- oder `LlmModel`-Rows vom Admin angelegt wurden, gehen diese beim Downgrade verloren. Vorher sichern mit:

## Deploy-Anmerkungen

- Dockerfile-CMD: `uvicorn app.main:app` — kein Alembic-Aufruf im Deploy.
- `init_db()` im FastAPI Startup-Event nutzt `Base.metadata.create_all` → neue Tabellen entstehen bei Redeploy.
- `init_db()` seedet `document_types` und `llm_models` idempotent (nur bei leerer Tabelle).
- `seed_llm_models` prüft SDK + API-Key pro Provider und seedet nicht-verfügbare Provider als `is_active=False`, damit sie nicht im Drawer-Dropdown erscheinen, bis SDKs/Keys eingerichtet sind.
- Nach Deploy `6991e9c` hat die Prod-DB 10 aktive OpenAI-Modelle, 15 inaktive Anthropic/Google-Modelle (kann via Admin-UI reaktiviert werden, sobald `anthropic`/`google-generativeai` installiert sind und die Keys gesetzt sind).
